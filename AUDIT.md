# Ревизия: связки, ключи, Telegram Mini App и путь «приложение для родителей»

## 1. Переменные окружения и секреты

### Фронт (Vite / Vercel)

| Переменная      | Где задаётся              | Назначение |
|-----------------|---------------------------|------------|
| `VITE_API_URL`  | Vercel → Settings → Env; локально `.env` | URL проекта Supabase (`https://xxxxx.supabase.co`). **Обязательно** при сборке: без неё запросы к API идут на пустой BASE (относительный путь), авторизация в Telegram не работает. |

**Проверка:** В Vercel в настройках проекта должны быть переменные окружения для Production (и при необходимости Preview) с ключом `VITE_API_URL` и значением Project URL из Supabase. Без этого в Mini App будет «Не настроен сервер» или ошибки при вызове `telegram-auth`.

### Supabase Edge Functions

| Секрет                 | Назначение |
|------------------------|------------|
| `TELEGRAM_BOT_TOKEN`   | Токен бота от @BotFather. Используется в `telegram-auth` для проверки `initData`. |
| `JWT_SECRET`           | Секрет для подписи JWT (один и тот же в `telegram-auth` и `diagnostic`). Рекомендуется `openssl rand -hex 32`. |

Задаются в Supabase: **Project Settings → Edge Functions → Secrets** (или `npx supabase secrets set`).

Автоматически доступны в Edge Functions (без явной настройки):

- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

Их подставляет сам Supabase при вызове функций.

## 2. Исправление в коде (уже внесено)

### `supabase/functions/telegram-auth/index.ts`

- **Проблема:** `validateInitData` использовала `await` внутри, но не была объявлена как `async`, что в рантайме приводило бы к ошибке и падению авторизации.
- **Исправление:** Функция объявлена как `async` и возвращает `Promise<Record<string, string> | null>`.

После деплоя Edge Functions (`npx supabase functions deploy`) авторизация по `initData` должна выполняться корректно.

## 3. Валидация initData (Telegram)

- Алгоритм в коде соответствует официальной документации:  
  `secret_key = HMAC_SHA256("WebAppData", bot_token)`, затем проверка `hash` через HMAC от data-check-string с этим секретом.
- Проверка `auth_date` (не старше 24 часов) есть.
- Для сторонней проверки без бот-токена в API есть поле `signature` (Ed25519); наш бэкенд использует классическую проверку по `hash` и бот-токену — этого достаточно.

## 4. Почему в Telegram мог появляться «Load failed»

Кратко по цепочке загрузки и типичным причинам:

1. **Что происходит при открытии Mini App**
   - Запрос идёт на `https://prof-miniapp.vercel.app/` (или другой URL из BotFather).
   - Сейчас: **middleware** по User-Agent «Telegram» отдаёт статичную страницу «Открыть в браузере» — тогда «Load failed» не должен показываться, пользователь видит заглушку.
   - Раньше (без middleware): отдавалась SPA (index.html → bootstrap → main.js → AppFull и т.д.). «Load failed» означал, что что-то в этой цепочке не загрузилось в webview.

2. **Типичные причины «Load failed» в webview Telegram (по документации и практике)**
   - Таймаут или сетевые ограничения при загрузке первого/тяжёлого JS.
   - Особенности Android (другая версия WebView, TLS, прокси/Cloudflare).
   - Ошибка при разборе/выполнении JS (синтаксис, неподдерживаемый API).
   - Отсутствие или неверный HTTPS, проблемы с сертификатами.

3. **Что уже сделано для снижения риска**
   - Минимальный первый скрипт (bootstrap ~2 KB), без React и без скрипта telegram.org в `index.html`.
   - Убран блокирующий запрос к `telegram.org/js/telegram-web-app.js` из HTML.
   - Error Boundary и отложенный вызов `readyTelegram()` в приложении.
   - Опционально: middleware с заглушкой для Telegram — пользователь видит «Открыть в браузере» вместо пустого «Load failed».

4. **Если снова пробовать грузить полное приложение в Telegram**
   - В BotFather указать URL с `?app=1`, чтобы middleware не срабатывал и отдавалась SPA.
   - Проверить в Vercel, что для этого деплоя задан `VITE_API_URL`.
   - Проверить в логах Supabase Edge Functions вызовы `telegram-auth` и возможные ошибки.

## 6. Путь «приложение для родителей» (последовательно)

Идея: сначала сделать рабочий сценарий для родителей, чтобы вся цепочка (бот → Mini App / веб → бэкенд) стабильно работала, затем переносить логику на детское приложение.

Предлагаемая последовательность:

1. **Отдельный бот для родителей** (или отдельное Mini App у того же бота).
   - В BotFather: новый бот или отдельный URL Mini App (например отдельный путь или поддомен для «родительского» приложения).

2. **Минимальное родительское приложение.**
   - Одна страница: приветствие, краткое описание («здесь будут отчёты по ребёнку»), кнопка «Открыть отчёт» (пока заглушка).
   - Тот же стек: React, тот же деплой на Vercel, те же Edge Functions (или отдельные функции для родителей).
   - Авторизация: тот же механизм (initData → `telegram-auth`), отдельная таблица `users` или флаг `role: 'parent'` / отдельная таблица `parents`.

3. **Связка родитель ↔ ребёнок.**
   - Таблица `parent_child` (parent_user_id, child_user_id, invite_code или status).
   - Инвайт: ребёнок в своём Mini App генерирует код/ссылку; родитель вводит код в своём приложении и привязывается.
   - Либо: родитель создаёт инвайт, ребёнок переходит по ссылке и подтверждает.

4. **Данные для родителей.**
   - Edge Function (например `parent-get-report`): по JWT родителя и привязке возвращает агрегированные результаты диагностики ребёнка (без лишних персональных полей).
   - В родительском приложении — экран «Результаты ребёнка» (только если привязка есть).

5. **После стабилизации.**
   - Включить/доработать детское приложение: те же ключи, тот же Vercel/Supabase, уже проверенный поток авторизации и загрузки. При необходимости убрать или изменить middleware для Telegram (например показывать заглушку только на отдельном URL).

Так мы сначала проверяем связки (ключи, секреты, initData, JWT, деплой) на простом сценарии для родителей, а затем возвращаемся к детскому приложению с уже работающей инфраструктурой.
